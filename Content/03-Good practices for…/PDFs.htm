PDFs

This section tells you all about PDFs and why it is better not to use them (but if you must, THIS IS THE WAY)!

Portable Document Format (PDF) files are complicated for assistive software and are not recommended in technical documentation or on web pages. If you can avoid them, do so.

However, if there is no other way than to use them, make sure to create them according to the best practices mentioned below.

Best practices for PDFs:

  • Create PDF files by using word processors (like Microsoft Word or PowerPoint). If you decide to use Microsoft Word, make sure to use the Accessibility checker option (Go to File > Info > Check Issues > Check Accessibility.)

  • Create meaningful links presenting their format and size (for example, TECH DOCS MAY, 5MB).

  • If possible, use PDF authoring tools such as Adobe Reading Order tool, Adobe Acrobat’s PDFMaker, Adobe FrameMaker, Adobe InDesign, CommonLook Office for Microsoft Office from Netcentric Technologies, LibreOffice, Lotus Symphony Documents, OpenOffice.org Writer.

  • Try not to use scans for your PDF content. However, if that is inevitable, try to scan materials with an Optical Character Recognition (OCR) scanner, which converts a scanned PDF text to an actual text file.

  • Tag a PDF file so that screen readers can read it. The method of creating a “tagged” PDF will be different depending on the original file format and operating system.

When creating Word or PowerPoint content, check WebAIM: Word and PowerPoint Accessibility Evaluation Checklist.

Credits: commonlook.com

If you do not have resources to create accessible tagged PDFs or their equivalents, you can use paid resources to repair your PDFs like Common Look, SensusAccess , or Adobe Acrobat Accessibility Guides.
