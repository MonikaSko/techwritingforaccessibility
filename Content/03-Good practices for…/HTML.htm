HTML

This section provides basic information about HTML, why it is good for accessibility, and how you can use it in your documentation.

HTML is, by default, accessible.

When using HTML, What You See Is What You Get (WYSIWYG). Simple as that.

It is recommended to make your HTML code as semantic as possible. Why?

Let’s analyze two ways of adding button to your documentation. You can use non-semantic option and add it like

Copy 
<div> Show documentation main page <div/>

or

Copy 
<button> Show documentation main page <button/>

Using the second option does not only add some default styling to your button but also has a built-in keyboard accessibility functionality. It allows users to navigate between all the buttons on the page while using the Tab key and make their selection by Return or Enter keys.

There is no such possibility with button placed between <div>.

Semantic HTML is also easier to be used for different devices like mobiles due to its lighter structure. It is also better for your SEO, as search engines give more attention to semantic attributes like <h1>, and keywords inside headings, than content between <div> tags.

 

Best practices for HTML:

  1. When you use semantic HTML screen readers, get all the information they need:

    Table created in W3C creator.

  2. Make sure that headings are correctly written and tagged.

  3. Keep the structure of the page:

    • Form main content (the one that is not repeated on your sites, like footer, navigation, and header) use <main> tag.

    • If you are adding navigational elements, place them under <nav> tag

    • Use <header> and <footer> tags for header and footer respectively

    • Remember that nested listed items are tagged with <il>

    • If you are using bullet list use <ul>

    • For numbered lists use <ol>

    • For each separate paragraph, use <p> tag

    • Add <lang> attribute to your main page to indicate the language of the page

  4. Remember about Alternative texts for images.

  5. Get familiar with Accessible Rich Internet Applications (ARIA) roles. They tell assistive technology users all about each interactive element or landmark that may not be clear enough to them. ARIA roles are best when used on non-semantic tags, such as <span> or <div>.

  6. Make sure to use landmarks. All elements on the page should be contained in a landmark element for example <main>, <aside> or <nav>. For HTML5 there is no need to define landmark elements, but when using HTML4 or ARIA pay special attention to them and define them for example as role=”main”.
